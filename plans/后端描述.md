📷 后端系统开发描述提示词 (Copy below)
角色设定：你是一名精通 Flask、MySQL 和 RESTful API 设计的后端工程师。

任务：请根据前端页面的数据需求和系统设计方案，构建“数字传承 · 红色德兴”心愿墙的后端系统描述与规范。

1. 整体架构与技术栈
核心框架：Flask (Python 3.10+)。
风格：RESTful API，返回标准 JSON 格式数据。
数据库：MySQL 8.0 (生产环境)。
ORM：PyMySQL。

1. 数据库设计 (Database Schema)
表名：`wishes` (心愿表)
设计要求：
- `id`: INT, 自增主键。
- `user_uid`: VARCHAR(64), 用户的唯一标识 (存储前端生成的 UUID)。
- `nickname`: VARCHAR(64), 用户昵称。
- `avatar`: VARCHAR(255), 用户头像 URL。
- `category`: VARCHAR(32), 心愿类别 (枚举值: 红色传承, 乡村建设, 产业发展, 生态环保)。
- `content`: TEXT, 心愿的具体内容 (最大 500 字)。
- `ai_response`: TEXT, “数字回响”内容 (后端调用 AI 生成或预设语料库匹配)。
- `likes`: INT, 点赞数, 默认为 0。
- `client_ip`: VARCHAR(45), 记录发布者 IP (用于简单的防刷限制)。
- `created_at`: DATETIME, 创建时间。

1. API 接口设计 (API Endpoints)
所有接口前缀：`/api`

(1) 获取心愿列表
- 方法：`GET /api/wishes`
- 参数：
  - `page` (int, default=1): 分页页码。
  - `per_page` (int, default=10): 每页数量。
- 响应：
  ```json
  {
    "total": 100,
    "wishes": [
      {
        "id": 1,
        "nickname": "张建国",
        "avatar": "...",
        "content": "愿红色基因...",
        "ai_response": "薪火相传...",
        "category": "红色传承",
        "created_at": "2026年1月22日",
        "likes": 88
      }
    ]
  }
  ```

(2) 发布心愿
- 方法：`POST /api/wishes`
- 请求体 (JSON)：
  ```json
  {
    "user_uid": "RedGuard_007",
    "nickname": "红脉卫士",
    "category": "乡村建设",
    "content": "希望家乡的路修得更好..."
  }
  ```
- 逻辑：
  1. 校验输入内容不为空。
  2. (可选) 进行敏感词过滤。
  3. 触发“数字回响”生成逻辑 (见第4点)。
  4. 存入数据库。
- 响应：返回新创建的心愿对象 (含生成的 ai_response)。

(3) 获取统计数据 (用于前端图表)
- 方法：`GET /api/stats`
- 响应：
  ```json
  {
    "pie_data": [
      {"name": "红色传承", "value": 120},
      {"name": "乡村建设", "value": 80}
    ],
    "word_cloud": [
      {"name": "德兴", "value": 50},
      {"name": "致富", "value": 30}
    ]
  }
  ```

4. 核心业务逻辑：AI“数字回响” (Digital Echo)
为了体现“科技激活文旅”的主题，后端需在用户提交心愿时生成一段回复。
实现策略 (分阶段)：
- 阶段一 (Mock/规则库)：准备一个包含 50 条红色文化语录的列表，根据用户选择的相关类别 (`category`) 随机抽取一条温暖的回复。例如：类别为“红色传承”时，返回“星星之火，可以燎原。您的心愿是传承的火种。”
- 阶段二 (接入 LLM)：调用通义千问/文心一言 API，Prompt: “用户是一个关注德兴发展的建设者，他许愿说：{content}。请以‘数字红色助手’的身份，用温暖、充满希望的语气写一句简短的回复（30字以内）。”

5. 安全与性能
- CORS 配置：允许前端域名/IP 跨域访问。
- 防抖与限流：同一 IP 在 1 分钟内只能发布 1 条心愿。
- 异常处理：统一捕获 500 错误并返回友好的 JSON 错误信息。


7. 新增数据表与接口（支持点赞、评论、删除）

为配合前端的社交交互，需要增加 `comments` 表并补充接口设计：

数据库新增表建议：

- 表名：`comments`
  - `id` INT AUTO_INCREMENT PRIMARY KEY
  - `wish_id` INT NOT NULL  -- 关联 `wishes.id`
  - `user_uid` VARCHAR(64)
  - `nickname` VARCHAR(64)
  - `avatar` VARCHAR(255)
  - `content` TEXT
  - `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP

新增 API 设计：

(4) 点赞（增/撤销）
- 方法：`POST /api/wishes/:id/like`
- 请求体（JSON）：{ "user_uid": "RedGuard_007", "action": "like" } // action 可为 like/unlike
- 响应：{ "id": 1, "likes": 123 }

(5) 添加评论
- 方法：`POST /api/wishes/:id/comments`
- 请求体：{ "user_uid": "RedGuard_007", "nickname": "张三", "avatar": "...", "content": "很好的想法" }
- 响应：返回新创建的评论对象

(6) 删除评论
- 方法：`DELETE /api/wishes/:id/comments/:comment_id`
- 权限：仅允许评论的 `user_uid`、心愿发布者或管理员删除
- 响应：{ "ok": true }

(7) 删除心愿
- 方法：`DELETE /api/wishes/:id`
- 权限：仅允许心愿发布者删除（验证 `user_uid`）或管理员
- 响应：{ "ok": true }

备注：实现这些接口时请注意：
- 对于点赞和评论的操作应返回最新的统计数据以便前端刷新展示。
- 对删除操作应做权限校验并在删除后更新关联统计（如评论计数）。
